# Ditto (Music Vibe) - 核心技術討論與工作分配模塊 (Technical Discussion & Work Assignment)

本文件盤點了專案進入開發階段前，必須共同確認的 **5 個核心技術底層架構**。

## 👥 團隊協作與分工指南 (Team Roles & Assignment)

因目前開發團隊由兩位專注於後端架構的工程師組成，專案採用**「垂直領域分工 (Vertical Domain Split)」**。前端與 UI 開發任務皆標示為 `[暫不分配]`，待未來前端工程師加入後再行指派。

*   **資料數據端**：
    負責系統的資料源頭，確保海量聽歌紀錄能穩定、高效地被抓取與查詢。主導第三方 API 串接、排程建置 (Cron Job)，以及多時間維度 (日/週/月/季/年) 的資料聚合演算法。
*   **配對與通訊端**：
    負責系統的核心商業邏輯，確保配對精準度與即時聊天的穩定性。主導每日配對演算法設計、OAuth 驗證，以及 WebSocket (Socket.io) 聊天室連線與防作弊機制建置。
*   **🌟 共同負責 (Database Schema)**：
    資料庫架構由雙方共同探討制定，確保資料源頭與消耗端的規格完美對接。

---

## 1. Database Schema (資料庫關聯表設計) 🌟 優先確認
資料庫是整個系統的基石，關係到後續 API 的開發順暢度。

*   **預計負責人**：**共同主導**，兩位後端需優先確定此 Schema 以利各自模組的平行開發。
*   **核心實體 (Entities)**：
    *   `User` (使用者基本資料與 OAuth 綁定狀態)
    *   `Track` (單曲 Metadata，如歌名、藝人、專輯封面圖片 URL)
    *   `Playlist` (交友歌單，記錄使用者自訂的 5 首精華)
    *   `MatchRoom` (配對狀態與 48 小時聊天室)
    *   `SyncLog` (串流平台背景抓取的聆聽日誌)
*   **🔑 關鍵決策點 (Decision Points)**：
    1.  **30 秒精華儲存機制**：使用者的「30 秒精華片段」該如何儲存？（建議方案：存 `start_time` 與 `end_time` 兩個 INT 欄位對應歌曲 ID）。
    2.  **歌單重複調用架構**：如何設計「歷史交友歌單」的關聯結構，讓使用者能快速重複套用過去選好的 5 首歌，同時避免資料庫產生大量冗餘紀錄？

## 2. 核心 API 架構：配對演算法與推薦機制 (Matching Algorithm)
負責決定使用者每天遇見誰，牽涉後端查詢效能與演算法邏輯。

*   **預計負責人**：**配對通訊端**
*   **🔑 關鍵決策點 (Decision Points)**：
    1.  **演算觸發時機**：每天派發的「5 位候選人」是系統在半夜利用排程 (Cron Job) 批次算好存起來？還是使用者打開交友大廳時才「即時 (On-the-fly)」運算派發？（考量伺服器尖峰負載與資料庫效能）。
    2.  **相似度權重邏輯**：當使用者選擇「相似曲風」時，系統的演算是比對雙方聆聽日誌的曲風標籤 (Tag) 重疊率？還是比對共同追蹤的藝人？演算的深度與精準度需明確定義。

## 3. 核心 API 架構：背景數據同步與時間維度聚合 (Data Sync & Aggregation)
負責處理串接第三方音樂 API，並生成龐大且彈性的數據圖表。

*   **預計負責人**：**資料數據端**
*   **🔑 關鍵決策點 (Decision Points)**：
    1.  **API 抓取頻率與限流防護**：我們多久向 Spotify / Apple Music 發送一次 Request 索取播放紀錄？如何設計 Retry 機制以避免觸發 Rate Limit (限流) 導致伺服器 IP 被鎖？
    2.  **多維度圖表聚合效能**：系統支援了高自由度的「日/週/月/季/年」時間切換。我們是否要在資料庫針對這些時間維度建立「關聯視圖 (Materialized Views)」或是每日排程算好寫入 Redis 緩存？以確保使用者切換「年度回顧」時，前端不會因為撈取龐大資料而卡頓。

## 4. 即時通訊架構：WebSocket 限時聊天室與同步聆聽 (Real-time Chat)
處理無延遲的 Socket 連線，以及特殊的限時倒數與深層連結機制。

*   **預計負責人**：後端 WebSocket Server 由**配對通訊端**負責。
    >前端狀態與 Deep Link 串接**：`[暫不分配]`。
*   **🔑 關鍵決策點 (Decision Points)**：
    1.  **48 小時逾時判定權責**：時間一到，是透過後端伺服器主動中斷 WebSocket 連線並推送「雙盲結算表單」指令，還是由前端計時器自己倒數並鎖定畫面發出 API 請求？（為防止竄改本地時間作弊，建議以後端時間戳記為準）。
    2.  **同步聆聽 Deep Link 實作**：雙方按下同步時，WebSocket 廣播的封包內容是什麼？（只需傳送如 `spotify:track:xxxx` URI 讓接收端呼叫手機原生系統開啟 App 即可？跨平台 iOS/Android 相容性需確認）。

## 5. 前端效能與演算法：單色渲染與泡泡物理引擎 (Frontend Performance)
挑戰極致的 UI/UX 動畫效能，確保高質感 Glassmorphism 的流暢度。

*   **預計負責人**：`[暫不分配 (留待未來前端負責)]`
*   **🔑 關鍵決策點 (Decision Points)**：
    1.  **單一專輯主色調抽取 (Dominant Color)**：這件事要在「前端」做（利用 Canvas `getImageData` 讀取圖片，可能耗損效能與遇到 CORS 跨域圖片問題），還是由「後端」在抓取音樂資料時，直接用函式庫算好 Hex 顏色碼存進資料庫，直接當作字串回傳給前端渲染？
    2.  **心智圖無重力泡泡引擎**：在不依賴龐大物理引擎框架（如 Matter.js）的前提下，漂浮與碰撞的數學邏輯該怎麼用最輕量的 Vanilla JS 與 CSS `requestAnimationFrame` 實作，並保持跨裝置 60fps 的順暢感？
