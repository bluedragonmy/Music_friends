# 音樂數據與限時品味交流社交 APP - 專案計畫書

本計畫書參酌 [Hahow：政府提案計畫書怎麼寫？](https://hahow.in/contents/articles/637effca6a8a4684e56a9cf4) 一文之核心觀念，以「功能導向」的工具屬性為出發點，將專案的「心臟（核心價值）」、「藍圖（架構圖）」與「動脈（流程化）」具象化，並以精確洗鍊的文字說明產品執行方向。

---

## 一、 計畫核心價值

*   **解決的核心問題：**
    1.  **紀錄聽音樂喜好數據的痛點**：使用者難以客觀且自動化地紀錄長期聆聽軌跡，缺乏有效的數據儀表板與日誌來統整自身在不同生活情境下的音樂品味。
    2.  **用音樂交友的破冰障礙與品味同溫層**：現代演算法造成單一推薦的音樂同溫層。且在傳統社交軟體中，人際互動往往因缺乏深刻的共同話題而難以破冰，無法單憑真實的音樂品味建立高黏著度的連結。
*   **預期創造價值：**
    打造一款結合「**音樂數據追蹤**」與「**限時品味交流**」的社交應用程式。提供一套**自動化紀錄品味**的工具，讓使用者可以客觀看見並分析自己的音樂輪廓。以此音樂喜好數據為基礎，利用限定時間與雙向驗證的機制，提供真實、無壓力且高品質的音樂社交配對體驗。

---

## 二、 計畫架構統整

將計畫拆解為四大核心要素，釐清執行邊界與牽涉之範圍。

### 1. 主題 (Topic)
開發一款結合「**音樂數據追蹤**」與「**限時社交互動**」雙核心的 APP，讓聆聽數據成為品味展示與社交配對的最佳資本。

### 2. 操作方向 (Operation Direction)
*   **音樂數據模組**：自動在背景同步使用者於串流平台的聆聽紀錄，支援日/週/月/季/年等多維度自由切換，並轉化為「心智圖」等高質感 Glassmorphism 視覺化數據儀表板。
*   **社交配對模組**：系統每日派發 5 位候選人。使用者可自訂 30 秒專屬片段，雙方透過直覺的右滑配對/左滑跳過機制快速評估是否交友，且無強制聆聽秒數限制。配對播放時，系統將依據歌曲抽出「單一專輯封面主色調」自動渲染全域背景，極致提升沉浸感。
*   **限時互動模組**：配對後開啟 48 小時倒數聊天室。在聊天室開啟期間，任一方皆可隨時發起「同步聆聽」喚醒本機第三方 App；而當 48 小時期限屆滿時，系統才會強制進行雙盲選擇結算。

### 3. 資源連結 (Resources)
*   **內部開發資源**：前端應用工程師（開發使用者介面與資料儀表板）、後端工程師（實作背景數據同步機制、WebSocket 限時聊天室）、UI/UX 設計師。
*   **外部 API 資源**：Apple Music API、Spotify Web API、YouTube Data API 授權與介接，以穩定獲取使用者的聆聽紀錄與時間戳記。

### 4. 利害關係人 (Stakeholders)
*   **目標受眾**：
    *   **傳統交友軟體疲勞者**：厭倦純看臉與無意義的破冰對話，渴望以「音樂品味」作為雙方第一印象，建立深層且真實靈魂契合的社交關係。
    *   **尋求情緒共鳴的傾聽者**：在特定情緒（如深夜失眠、低潮）下尋求能懂這個氣氛的陪伴者，藉由 48 小時限時聊天與同步聆聽，獲得無壓力且具隱私的短期靈魂共鳴。
*   **資料服務供應方**：各大串流媒體平台與授權合作夥伴。
*   **內部維運團隊**：負責產品伺服器維護與用戶反饋迭代。

## 三、 執行步驟與時間軸

將抽象想法轉換為具可執行性的時程規劃，推動計畫血液流動。

*   **Phase 1：需求收斂與基礎資料庫設計**
    *   確定 PRD (產品需求文件) 與 User Story 細節。
    *   設計用戶設定檔、聆聽歷史紀錄與配對日誌等核心資料庫 Schema。
*   **Phase 2：音樂日誌與數據追蹤模組開發**
    *   建立 OAuth 驗證機制，並完成音樂平台 API 對接。
    *   實作「背景數據自動同步機制」，定期抓取播放紀錄並生成時段與曲風之交叉分析圖表。
*   **Phase 3：交友演算法與配對功能實作**
    *   根據聆聽總體數據庫，開發交友配對系統。
    *   建立前台的每日歌單自訂系統、具備佇列與切換功能的播放介面，並實作「單一專輯封面主色調」動態背景渲染，以及自由無限制的卡片滑動 (Swipe) 配對機制。
*   **Phase 4：社交聊天室模組與測試**
    *   WebSocket 聊天室實作，包含 48 小時倒數計時、雙盲結算邏輯、歌單快照調閱與「同步播放」功能。
    *   針對串接 API 規劃與測試合理的請求頻率防禦。
*   **Phase 5：封測驗證與公測發布**
    *   啟動 Beta 測試蒐集目標受眾針對「數據紀錄」與「社交配對」之綜合反饋。
    *   優化數據與伺服器效能，正式公開營運。

## 四、 關鍵效益與預期風險控管

文字需精確說明可能風險與應對方針，確保企劃邏輯嚴謹可行。

1.  **穩定的第三方 API 介接與防護**
    專案高度依賴 Spotify 等外部 API 同步聽歌紀錄。系統將設計背景批次處理架構，定期將獲取之資料落庫於系統本地伺服器。此舉不僅能安全儲存使用者的珍貴歷史紀錄，更能減輕對第三方 API 的頻繁調用依賴，防止超過限流 (Rate Limit) 限制造成之系統中斷。
2.  **無版權侵權風險**
    系統僅紀錄與運算歌曲 Metadata（如歌名、藝人、時間戳記）。在社交配對環節中，所有的「同步聆聽」與「音檔播放」皆交由使用者原生安裝之第三方 App (如 YouTube Music 或 Spotify) 執行，徹底避開音檔下載與授權費用等侵權疑慮。
3.  **交友與社交隱私保護機制**
    運用 48 小時「限時對話」與期滿後的「雙盲結算」機制，建立一套天然的交友過濾網。從系統面大幅減輕用戶交流過程的心理負擔；同時常駐之封鎖與檢舉功能能充分保障交友安全性，形成高品質的純粹音樂社交生態圈。
